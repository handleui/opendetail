# FAQ

## Does opendetail ship a chat UI?

No. `opendetail` is the engine. The current package focuses on the build step, local index, runtime, and Next.js adapter. If you want a chat surface, you bring that separately.

## Is this built on OpenAI?

Yes. `opendetail` uses the OpenAI Responses API underneath. The product value is the setup flow, grounding model, local retrieval layer, and cleaner integration surface around that API.

## Do I need a vector database or external search service?

No. Today the default flow builds a local JSON index and searches it locally. That keeps the setup small and easy to reason about.

## What content gets indexed?

Whatever matches the `include` globs in `opendetail.toml`, minus anything in `exclude`. The main target is Markdown and MDX content that already explains your product clearly.

## Does MDX still work if the file uses components?

Yes, but the important part is the text. `opendetail` keeps useful MDX text and headings; imports, exports, and MDX-only syntax are not the point of retrieval.

## Can it answer from outside my docs?

No. The runtime is designed to answer only from the retrieved sources. It should cite those sources instead of filling gaps with general knowledge.

## What happens when the docs are not enough?

It falls back with a plain answer: `I couldn't find that in the configured docs.` That behavior is intentional. The product is more useful when it fails clearly than when it improvises.

## Where does the data live?

The built index lives in your app at `.opendetail/index.json`. At runtime, `opendetail` loads that local index and sends only the retrieved sections to the model. In `self-hosted`, that means your runtime needs `OPENAI_API_KEY`. In `hosted`, the client transport points at a hosted endpoint instead.

## What are the integration modes?

There are two documented modes:

- `self-hosted`: scaffold a route and keep `OPENAI_API_KEY` in your own runtime
- `hosted`: skip route scaffolding and point your client transport at a hosted endpoint later

The request and stream contract stay the same across both modes.

## Is the hosted backend available yet?

No. In this phase, the hosted path is documented and scaffolded only. The OpenDetail-managed proxy backend is not shipped yet.

## What frameworks are supported today?

Today the package ships a core Node runtime and a thin Next.js adapter via `opendetail/next`. That means Next.js is the first framework-specific path right now, while the runtime itself stays framework-light.

## Is setup still manual?

The CLI includes a setup flow (`opendetail setup`) that can scaffold config, route wiring for `self-hosted`, and an initial index. It also includes a diagnostics command (`opendetail doctor`) for fast verification in both integration modes.

## Is it only for developer docs?

No. Developer docs are a natural fit, but the shape also works for product guides, onboarding, help content, and internal knowledge. If the content is in Markdown and MDX and answers real questions, it can fit.

## Does it replace docs?

No. `opendetail` works because the docs or knowledge content already exist. It makes that material easier to ask, retrieve, and cite inside an app.
