# How It Works

`opendetail` follows a small two-part flow: build once from your Markdown and MDX, then answer questions from the resulting local index. The design stays simple on purpose so the product remains easy to reason about and easy to inspect.

## Build time

You can start with `opendetail setup`, which scaffolds the defaults most teams need. In `self-hosted` mode, that means config, route file, and initial index. In `hosted` mode, it means config and initial index, while skipping the route file on purpose. Under the hood, both flows still center on `opendetail.toml`, where you define which Markdown and MDX files to include, which ones to exclude, and which base path should be used when source URLs are generated.

When you run the build command, `opendetail` reads the matched files and turns them into a local index at `.opendetail/index.json`. Content is split by heading sections so the runtime can retrieve smaller, more relevant pieces instead of entire files. The output is local and file-based, which keeps the setup light and makes the index easy to inspect.

## Request time

At request time, the runtime loads the local index into memory and searches it for the most relevant sections. Today that retrieval step is lexical and local. There is no separate vector database, sync layer, or external search service in the default flow.

Once the relevant sections are selected, `opendetail` sends only those sections to the OpenAI Responses API along with strict instructions: answer only from the provided sources, cite factual claims, and do not invent missing information. That is what makes it a grounded assistant rather than a general chat layer.

## Runtime shape

The current package is server-first. You can use the core runtime directly in Node, or use the thin Next.js adapter that exposes a route handler with a one-line `createNextRoute()` helper. In Next.js, that route must run on the Node.js runtime.

Responses can be streamed, and the response shape includes source information alongside the generated answer. The request shape stays `{ question: string }`, and the stream contract stays `meta`, `sources`, `images`, `delta`, `done`, `error` across both integration modes. If the configured docs do not contain enough information, `opendetail` falls back instead of pretending it knows more than the content says.

## Integration split

`self-hosted` is the current end-to-end backend path. You scaffold a route, keep `OPENAI_API_KEY` in your own runtime, and let your server call the model.

`hosted` is the setup-focused path. In this phase, it is documented and scaffolded, but the OpenDetail-managed proxy backend is not shipped yet. The point of the split is to make that future setup path explicit now without changing the grounding model or the request contract later.

## What this design buys you

This design keeps the product small, inspectable, and honest. Your content stays the source of truth. The local index stays close to the app. The answer path stays narrow enough that a reader can understand where the response came from and why.
