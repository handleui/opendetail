## v0.8.0 — Path-based setup and cleaner package boundaries

### Setup

OpenDetail now explains adoption through **Fastest**, **Branded**, and **Headless**. The CLI matches that model with mode-first setup, a semantic **doctor**, and a new **verify** command for structural checks and optional live NDJSON validation.

### Packages

The NDJSON transport now lives in **`opendetail-client`**. **`opendetail-react`** is React-only, and **`opendetail-next`** stays focused on the Next.js route adapter and link helpers.

### Hardening

Public surfaces are tighter too: safer source-link handling, interrupted responses that no longer look complete, no third-party favicon lookups in the default sources UI, and an early request-size guard in the Next.js handler.

**Changes:** Added **`opendetail-client`**; rewrote docs around the adoption-path model; tightened link safety and privacy defaults in the React and Next integrations.

## v0.5.0 — Docs index and streaming UX

### Build output

The CLI writes a single machine-readable **index.json** under `.opendetail`, so apps can load chunks for lexical search without a separate database. Paths stay stable across builds when content hashes do not change.

### Assistant

The sidebar assistant accepts streamed NDJSON (text, sources, images) so the UI can render incrementally. Known doc URLs are passed through for safe **source links** in Fumadocs-style layouts.

**Changes:** Stricter validation for remote tool definitions; clearer errors when the index file is missing or out of date; minor CSS fixes for the input surface on small viewports.

## v0.4.0 — CLI quickstart and configuration

**opendetail setup** scaffolds config, route, and initial index, with flags for base path, include globs, and optional media. **opendetail doctor** checks binary, config, and route wiring before you ship.

**Changes:** Configuration fields documented; improved progress output during **opendetail build**.
