OpenDetail turns the Markdown and MDX you already ship into **grounded answers inside your product**: same content your team edits, retrieved with citations instead of vibes. It is intentionally small: a build step, a local index file, a narrow HTTP contract, and thin adapters — not a second CMS and not a generic chat platform.

From your app root, the CLI can scaffold config, an optional Next.js route, and your first index build:

<CopyCommand bun="bunx opendetail setup" npm="npx opendetail setup" />

## How the pieces connect

Your docs and media live in the repo. **Build** turns matched files into `.opendetail/index.json`. When someone asks a question, your **route or client** loads that index, picks the best chunks with lexical search, and calls the OpenAI Responses API using only that context — plus any remote tools you configured. The UI receives streamed NDJSON (text, sources, images) so you can render like a live dev tool instead of blocking on a full response.

The same pages stay readable as normal docs. Machine-readable routes (for example `/llms.mdx/docs`) stay available for tooling without maintaining a parallel format.

## Who it is for

- **Product and devrel teams** who want in-app help without training a separate knowledge base.
- **Developers** who want a boring, testable pipeline: build artifact in git, predictable API, clear errors.
- **Agents** that need stable URLs and structured retrieval — the index and routes are built for that use case too.

## Packages (at a glance)

| Package | Role |
| --- | --- |
| `opendetail` | Build, core runtime, CLI (`opendetail` binary). |
| `opendetail-next` | Next.js route helpers for self-hosted mode. |
| `opendetail-react` | `useOpenDetail` and streaming client for browser UI. |
| `opendetail-fumadocs` | Optional helpers when you already use Fumadocs-style docs. |
