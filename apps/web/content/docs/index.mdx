OpenDetail turns the Markdown and MDX you already ship into **grounded answers inside your product**: same content your team edits, retrieved with citations instead of vibes. It is intentionally small: a build step, a local index file, a narrow HTTP contract, and thin adapters — not a second CMS and not a generic chat platform.

<CopyCommand bun="bunx opendetail setup" npm="npx opendetail setup" />

## How it fits together

| Stage | What happens |
| --- | --- |
| **Build** | Matched docs become **`.opendetail/index.json`**. |
| **Request** | Your route or client loads the index, runs lexical search, calls the Responses API with that context. |
| **UI** | Streamed NDJSON (text, sources, images) — you render with **your** components or **`opendetail-react`**. |

Docs stay human-readable on the web; machine-readable routes (for example `/llms.mdx/docs`) remain available for tooling.

## Who it is for

| Audience | Why |
| --- | --- |
| **Product & devrel** | In-app help without a separate knowledge base. |
| **Engineering** | Testable pipeline: artifact in git, predictable API, clear errors. |
| **Agents** | Stable URLs and structured retrieval from the same index. |

## Packages (at a glance)

| Package | Role |
| --- | --- |
| [`opendetail`](/docs/core) | Build, core runtime, CLI. |
| [`opendetail-next`](/docs/next) | Next.js App Router route, NDJSON streaming, `renderNextSourceLink`. |
| [`opendetail-react`](/docs/react) | `useOpenDetail`, `AssistantSidebar` (`connection` for streaming), streaming client. |
| [`opendetail-fumadocs`](/docs/fumadocs) | Optional: known-page resolver + `FumadocsAssistant`. |
| [`trifold`](https://github.com/handleui/opendetail/tree/main/packages/trifold) | Optional: parallel columns for small viewports. |

Layering and naming details: [**Packages & composition**](/docs/packages).

**Styling:** systems, themes, and primitives are covered in [**Styling**](/docs/design-system). **Component reference** (live previews) is under [**UI sandbox**](/ui).
