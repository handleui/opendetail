# Overview

`opendetail` turns your Markdown and MDX docs into a small, grounded assistant for your app.

It sits on top of the OpenAI Responses API, but the value is not a new agent runtime. The value is a smaller setup path, grounded retrieval from your own docs, and a cleaner integration surface than wiring that stack by hand.

## What OpenDetail is for

OpenDetail fits best when the hard part is not "how do we make a chatbot," but "how do we ship answers that stay tied to our own content."

It works well for:

- product help and onboarding
- developer docs
- internal guides
- feature explainers
- support content that already lives in Markdown or MDX

## What ships today

- a setup-focused CLI
- a local build step that writes `.opendetail/index.json`
- a core Node runtime
- a thin Next.js adapter for self-hosted routes
- installable client and React hook foundations in the registry

## Integration modes

| Mode | What you manage | Best for | Status |
| --- | --- | --- | --- |
| `self-hosted` | Your route, runtime env, and `OPENAI_API_KEY` | Private docs, server-owned control, simple default path | Available today |
| `hosted` | Your built index and client transport config | Faster setup and future hosted delivery | Documented and scaffolded in this phase |

## Why teams use it

- Answers stay tied to your own docs
- The runtime stays small and inspectable
- The request contract stays narrow: `{ question: string }`
- The stream contract stays stable across modes
- Missing documentation fails clearly instead of sounding confident

## Human and agent friendly

The docs are organized for people first: task-based quickstarts up front, reference pages after that, and short, direct explanations throughout.

They are also exposed through the Fumadocs machine-readable routes already wired into this app, which means agents can consume the same source material without needing a separate docs format.
