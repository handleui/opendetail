---
title: Fumadocs
description: Optional citation-link helpers for docs sites with known page URLs.
---

Use **`opendetail-fumadocs`** when your docs site already has predictable page URLs and you want OpenDetail citations to stay inside that known surface. It is optional glue, not a separate runtime.

That makes it most useful in **Fastest** and **Branded** setups where the assistant should cite pages under stable `/docs/...` routes.

## What it does

- Resolves known local page paths into safe citation targets
- Helps turn source references into links your docs site already understands
- Works with `renderNextSourceLink` from [`opendetail-next`](/docs/next) when you want Next.js link handling

## What it does not do

- It does not change how OpenDetail indexes content
- It does not change the backend contract
- It does not replace `opendetail-react`, `opendetail-next`, or `opendetail`

## API

- `resolveFumadocsSourceTarget` turns a source and a known page list into a safe link target or `null`
- `createFumadocsSourceTargetResolver(knownPageUrls)` returns a resolver you can pass through your assistant wiring
- `getFumadocsPageUrls(source)` extracts the set of page URLs from a Fumadocs-style source tree

## Convenience wrapper

`FumadocsAssistant` is the ergonomic wrapper when you already know you want the allowlist behavior. It bundles the assistant wiring with the citation resolver so you do not have to thread those pieces together by hand.

Use it when the docs URLs are stable and you want the assistant to stay inside that known map. If your site structure is more dynamic, wire the resolver directly instead.

## Wiring without the wrapper

The direct path is still simple: collect the page URLs your site actually serves, create the resolver, and pass that into the assistant wiring.

Local URLs must match known page paths. Unknown paths resolve to `null` so the UI does not invent links.
