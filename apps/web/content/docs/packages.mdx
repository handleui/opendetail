OpenDetail is intentionally **layered**: core runtime and build, then framework adapters, then UI. You pick the layers your app needs; they **compose** in one codebase.

## Layers (bottom to top)

1. **`opendetail`** — CLI, config, **`opendetail build`**, **`.opendetail/index.json`**, and the core runtime that loads the index and talks to the model. **Start here** for any integration.
2. **`opendetail-next`** — **Next.js App Router**: **`createNextRoute` / `createNextRouteHandler`**, NDJSON streaming, and **`renderNextSourceLink`** so assistant citations can render as **`next/link`** where appropriate.
3. **`opendetail-react`** — Browser UI: **`useOpenDetail`**, **`createOpenDetailClient`**, **`AssistantSidebarShell`**, and related assistant UI. **Framework-agnostic** (works with Vite, etc.) as long as you point the client at your HTTP endpoint.
4. **`opendetail-fumadocs`** (optional) — A **small preset** on top of **`opendetail-react`** + **`opendetail-next`**: **`createFumadocsSourceTargetResolver`**, **`FumadocsAssistant`**, and helpers for **known doc URLs**. The name reflects common **Fumadocs-style** `/docs/...` routes; the code does **not** import Fumadocs packages — it is an **allowlisted pathname resolver** you can use for any Next docs site that lists its routes.

**`trifold`** (optional, [separate package](https://github.com/handleui/opendetail/tree/main/packages/trifold)) — **Parallel columns** for small viewports (**`leading` · `center` · `trailing`** geometry). The OpenDetail **web** demo uses it inside **`renderMobileShell`**; it is **not** required for retrieval or streaming.

## When to use which package

| Package | Use when |
| --- | --- |
| **`opendetail`** | You need the build, CLI, runtime, or self-hosted index file. |
| **`opendetail-next`** | You deploy on **Next.js App Router** and want the official route handler + **`renderNextSourceLink`**. |
| **`opendetail-react`** | You build the **in-browser** assistant (hook, shell, streaming client). |
| **`opendetail-fumadocs`** | You want a **known-page allowlist** for citation URLs and/or the **`FumadocsAssistant`** wrapper instead of wiring **`resolveSourceTarget`** by hand. Skip it if you implement your own resolver (e.g. internal tools, multiple origins). |
| **`trifold`** | You want the **parallel-column** mobile shell UX (`Trifold` / `ParallelTrack`); purely presentational. |

## How they compound

Typical **Next.js + self-hosted** app:

- **`opendetail`** (build + runtime) +
- **`opendetail-next`** (route + `renderNextSourceLink`) +
- **`opendetail-react`** (`useOpenDetail` + `AssistantSidebarShell`).

Optional additions:

- **`opendetail-fumadocs`** — same stack, plus **`createFumadocsSourceTargetResolver([...])`** passed into the shell (or use **`FumadocsAssistant`**, which bundles resolver + link renderer + shell — see [opendetail-fumadocs](/docs/fumadocs)).
- **`trifold`** — only the **mobile shell** layout; compose via **`renderMobileShell`** on the assistant shell, not inside core packages.

There is **one** assistant **pipeline** (question → NDJSON → UI). **`FumadocsAssistant`** is **not** a second product — it is **`AssistantSidebarShell`** with Next-aware source links and a preset resolver.

## Naming in this monorepo

| Name | Role |
| --- | --- |
| **`AssistantSidebarShell`** | **`opendetail-react`** — slots for navigation, main content, and the assistant column. |
| **`FumadocsAssistant`** | **`opendetail-fumadocs/assistant`** — convenience wrapper: shell + **`renderNextSourceLink`** + allowlist resolver. |
| **`Sidebar`** | **`apps/web` only** — marketing site nav (not published as a package). |

If you outgrow the allowlist (e.g. dynamic internal URLs), keep **`opendetail-react`** + **`opendetail-next`** and supply your own **`resolveSourceTarget`**; you do not need **`opendetail-fumadocs`**.
