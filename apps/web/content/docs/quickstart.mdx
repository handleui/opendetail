## Scaffold

Generate `opendetail.toml`, the default API route, and an initial index:

<CopyCommand bun="bunx opendetail setup --with-media" npm="npx opendetail setup --with-media" />

What that writes:

- **`opendetail.toml`** — content globs and public path mapping
- **`src/app/api/opendetail/route.ts`** — the self-hosted route handler
- **`.opendetail/index.json`** — the built index (unless you pass `--skip-build`)

Add **`OPENAI_API_KEY`** to the environment your route runs in.

### Config shape

<CodeBlock lang="toml">{`version = 1

[[content]]
include = ["content/**/*.{md,mdx}"]
exclude = []
public_path = "/docs"

[media]
include = ["content/**/*.{png,jpg,jpeg,webp,avif,gif,svg}"]
exclude = []
public_path = "/content-media"
`}</CodeBlock>

Tweak **`[[content]]`** globs and **`public_path`** so each root matches where those files are served. Add more **`[[content]]`** blocks when you index multiple trees (for example `/docs` and `/`). Full field reference: [Core](/docs/core#fields).

## Integration modes

### Self-hosted (default)

Your Next.js route, your Node runtime, your API key, your rebuilt index on deploy.

| Requirement | Notes |
| --- | --- |
| Node.js | 18+ |
| Runtime | **Node.js** (not Edge) for the default adapter |
| Secrets | **`OPENAI_API_KEY`** where the handler runs |
| Artifact | **`.opendetail/index.json`** after `opendetail build` |

**Request body:**

<CodeBlock lang="json">{`{ "question": "How do I configure public_path?" }`}</CodeBlock>

**Response** — `application/x-ndjson; charset=utf-8`, events in order:

| Event | Role |
| --- | --- |
| `meta` | Model metadata |
| `sources` | Matched chunks (and remote citations when configured) |
| `images` | Grounded image references from matched docs |
| `delta` | Streamed answer text |
| `done` | Final text |
| `error` | Safe public error payload |

Malformed bodies return a JSON error. Initialization failures before streaming starts return JSON instead of NDJSON. In production, repeated init failures are cached.

### Hosted-shaped (scaffold only)

`--integration hosted` does not generate a route. It writes config and builds the index, but your app calls a remote endpoint you configure. The OpenDetail-managed cloud backend is not shipped yet.

<CopyCommand bun="bunx opendetail setup --integration hosted" npm="npx opendetail setup --integration hosted" />

Point the client at your endpoint:

<CodeBlock
  lang="ts"
  code={`import { useOpenDetail } from "opendetail-react";

const assistant = useOpenDetail({
  transport: {
    endpoint: process.env.NEXT_PUBLIC_OPENDETAIL_ENDPOINT,
    headers: process.env.NEXT_PUBLIC_OPENDETAIL_AUTH_TOKEN
      ? { authorization: \`Bearer \${process.env.NEXT_PUBLIC_OPENDETAIL_AUTH_TOKEN}\` }
      : undefined,
  },
});`}
/>

Request and stream events are identical to self-hosted — no breaking migration later.

## How grounding works

1. **Build time** — Matched files are parsed, split into heading-based chunks, written to `.opendetail/index.json`. No vector database required.
2. **Request time** — The runtime runs **lexical search**, sends the best chunks to the **Responses API**, optionally attaches **remote tools** from config.
3. **Instructions** — The model answers from those sources, cites factual statements, avoids inventing coverage, and says so when a topic is not in the docs.

If nothing matches, `fallback: true` is set on the response. The model may still answer using configured `fetch` tools; if it returns no usable text, the user sees a clear "not documented yet" message. See [fallback behavior](/docs/core#fallback-behavior).

## FAQ

| Topic | Answer |
| --- | --- |
| **No vector DB?** | Correct — the default path uses lexical search against a local index file. |
| **MDX with components?** | Text and headings are the retrieval unit; runtime component rendering is not. |
| **Hosted backend?** | Client scaffolding exists; managed backend is not shipped in this phase. |
