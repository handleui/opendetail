Use this page to get to a working setup quickly. The three adoption paths are:

- **Fastest**: OpenDetail sets up the route, config, and first build.
- **Branded**: you keep the backend contract and build your own UI.
- **Headless**: you use OpenDetail as the docs backend and connect your own client later.

**Fastest**

<CopyCommand bun="bun add opendetail opendetail-next opendetail-react" npm="npm i opendetail opendetail-next opendetail-react" />

**Branded**

<CopyCommand bun="bun add opendetail opendetail-next opendetail-client" npm="npm i opendetail opendetail-next opendetail-client" />

## Setup

Use the mode that matches your integration style:

<CopyCommand bun="bunx opendetail setup --mode fastest" npm="npx opendetail setup --mode fastest" />
<CopyCommand bun="bunx opendetail setup --mode branded" npm="npx opendetail setup --mode branded" />
<CopyCommand bun="bunx opendetail setup --mode headless" npm="npx opendetail setup --mode headless" />

What setup writes depends on the mode:

- **Fastest** writes `opendetail.toml`, the default route, and the first index build.
- **Branded** writes the same backend pieces, but you are expected to own the UI.
- **Headless** writes the config and index, without assuming a local UI route.

Add `OPENAI_API_KEY` to the environment where the route runs.

### Config shape

<CodeBlock lang="toml">{`version = 1

[[content]]
include = ["content/**/*.{md,mdx}"]
exclude = []
public_path = "/docs"

[media]
include = ["content/**/*.{png,jpg,jpeg,webp,avif,gif,svg}"]
exclude = []
public_path = "/content-media"
`}</CodeBlock>

Match each `[[content]]` block to where those files are served. Add more blocks when you index more than one tree, such as `/docs` and `/`. See [Core](/docs/core#fields) for the full field reference.

## Build, doctor, verify

<CopyCommand bun="bunx opendetail build" npm="npx opendetail build" />

Run `build` whenever docs change or before deploy.

<CopyCommand bun="bunx opendetail doctor" npm="npx opendetail doctor" />

Use `doctor` when the setup looks wrong or a required file is missing.

<CopyCommand bun="bunx opendetail verify" npm="npx opendetail verify" />

Use `verify` when you want a semantic setup check. Add `--endpoint` to exercise a live NDJSON stream end to end.

Full command and flag reference: [CLI](/docs/cli).

## How the paths differ

### Fastest

OpenDetail owns the docs route and the stream contract. You use the generated route and the built index.

### Branded

You keep the same backend contract, but the UI is yours. The easiest custom UI path is to use `opendetail-client` for transport and wire the stream into your own React state.

### Headless

You only want the docs backend and the build artifact. Another app or service can call the stream later.

## Request and response

The request body is simple:

<CodeBlock lang="json">{`{ "question": "How do I configure public_path?" }`}</CodeBlock>

The response is `application/x-ndjson; charset=utf-8` and the events arrive in this order:

| Event | Role |
| --- | --- |
| `meta` | Model metadata |
| `sources` | Matched chunks and remote citations when configured |
| `images` | Grounded image references from matched docs |
| `delta` | Streamed answer text |
| `done` | Final text |
| `error` | Safe public error payload |

Malformed bodies return JSON errors. Initialization failures before streaming starts also return JSON instead of NDJSON. Repeated init failures are cached in production.

---

## How grounding works

1. Build time parses matched files, splits them into heading-based chunks, and writes `.opendetail/index.json`. No vector database is required.
2. Request time runs lexical search, sends the best chunks to the Responses API, and can attach remote tools from config.
3. Answer time uses those sources, cites factual statements, and says when a topic is not in the docs.

If nothing matches, `fallback: true` is set on the response. The model may still answer using configured `fetch` tools; if it returns no usable text, the user sees a clear "not documented yet" message. See [fallback behavior](/docs/core#fallback-behavior).
