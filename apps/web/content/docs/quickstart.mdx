Follow this page in order once; after that, rebuild whenever docs change.

## Install

Add the packages you need for a Next.js app with the default route adapter:

<CopyCommand
  bun="bun add opendetail opendetail-next opendetail-react"
  npm="npm i opendetail opendetail-next opendetail-react"
/>

Use the copy control to cycle package managers (`npm i`, Yarn, pnpm, or Bun) with the same dependency list. Check that the CLI is available without a global install:

<CopyCommand bun="bunx opendetail --version" npm="npx opendetail --version" />

## Scaffold

Generate `opendetail.toml`, the default API route, and the first index (unless you skip the build):

<CopyCommand bun="bunx opendetail setup --with-media" npm="npx opendetail setup --with-media" />

What that can do:

- write **`opendetail.toml`**
- scaffold **`src/app/api/opendetail/route.ts`** (self-hosted default)
- run an initial build to produce **`.opendetail/index.json`**

Add **`OPENAI_API_KEY`** to the environment your route runs in before you call the API.

### Default config shape

You should see something like:

<CodeBlock lang="toml">{`version = 1

[[content]]
include = ["content/**/*.{md,mdx}"]
exclude = []
public_path = "/docs"

[media]
include = ["content/**/*.{png,jpg,jpeg,webp,avif,gif,svg}"]
exclude = []
public_path = "/content-media"
`}</CodeBlock>

Tweak **`[[content]]`** globs and **`public_path`** so each root matches where those files are served on the site. Add more **`[[content]]`** rows when you index multiple trees (for example `/docs` and `/`). Full field list is in [Core package](/docs/core#fields).

## Rebuild when content changes

Any time Markdown, MDX, or matched media changes:

<CopyCommand bun="bunx opendetail build" npm="npx opendetail build" />

Run that in CI before deploy, or as a `postbuild` step, so production always matches the docs you shipped.

**Diagnose** when something looks wrong:

<CopyCommand bun="bunx opendetail doctor" npm="npx opendetail doctor" />

<CopyCommand bun="bunx opendetail doctor --integration hosted" npm="npx opendetail doctor --integration hosted" />

CLI details: [CLI quickstart](/docs/cli/quickstart) · [CLI reference](/docs/cli/commands).

## Integration modes

### Self-hosted (default)

**Self-hosted** means: your Next.js route, your Node runtime, your API key, your rebuilt index on deploy.

The App Router integration lives in **`opendetail-next`**: **`createNextRoute`**, **`createNextRouteHandler`**, and **`renderNextSourceLink`**. See [**opendetail-next**](/docs/next) for exports, options, and requirements. Package layering: [**Packages & composition**](/docs/packages).

| Requirement | Notes |
| --- | --- |
| Node.js | 18+ |
| Runtime | **Node.js** for the default adapter (not Edge) |
| Secrets | **`OPENAI_API_KEY`** where the handler runs |
| Artifact | **`.opendetail/index.json`** after [`build`](/docs/cli/build) |

**Request body** — intentionally tiny:

<CodeBlock lang="json">{`{ "question": "How do I configure public_path?" }`}</CodeBlock>

Malformed bodies get a JSON error instead of a broken stream.

**Response** — `application/x-ndjson; charset=utf-8`, events in this order:

| Event | Role |
| --- | --- |
| `meta` | Model metadata |
| `sources` | Matched chunks (and remote citations when configured) |
| `images` | Grounded image references from matched docs |
| `delta` | Streamed answer text |
| `done` | Final text |
| `error` | Safe public error payload |

If initialization fails before streaming starts, you get JSON instead of NDJSON. In production, repeated init failures are cached so you do not pay the same failure cost on every request.

### Hosted-shaped (scaffold only)

**Hosted** mode in the CLI **does not** generate a route. It still writes config and builds **`.opendetail/index.json`**, but your app is expected to call a remote endpoint you configure — the OpenDetail-managed cloud backend is **not** shipped yet.

<CopyCommand bun="bunx opendetail setup --integration hosted" npm="npx opendetail setup --integration hosted" />

Point the client at your future endpoint with environment variables:

| Variable | Typical use |
| --- | --- |
| **`OPENDETAIL_ENDPOINT`** | Server-side URL |
| **`OPENDETAIL_AUTH_TOKEN`** | Optional bearer token |
| **`NEXT_PUBLIC_OPENDETAIL_*`** | Browser examples |

<CodeBlock
  lang="ts"
  code={`import { useOpenDetail } from "opendetail-react";

const endpoint = process.env.NEXT_PUBLIC_OPENDETAIL_ENDPOINT;
const authToken = process.env.NEXT_PUBLIC_OPENDETAIL_AUTH_TOKEN;

const assistant = useOpenDetail({
  transport: {
    endpoint,
    headers: authToken
      ? {
          authorization: \`Bearer \${authToken}\`,
        }
      : undefined,
  },
});`}
/>

Request and stream events stay the same as self-hosted so you do not need a breaking migration later.

## Grounding (short)

1. **Build time** — Matched files are parsed, split into heading-based chunks, and written to **`.opendetail/index.json`**. No required vector database in the default path.
2. **Request time** — The runtime loads the index, runs **lexical search**, sends the best chunks to the **Responses API**, and optionally attaches **remote tools** from config.
3. **Instructions** — The model is told to answer from those sources, cite factual statements, avoid inventing coverage, and admit when the topic is not in the docs.

If nothing matches well, you still get a response, but it may be marked as fallback; if the model produces no usable text, users see a clear “not documented yet” style message — see [fallback](/docs/core#fallback-behavior).

## FAQ

| Topic | Answer |
| --- | --- |
| **Chat UI?** | OpenDetail is the engine and contracts; you bring the UI — or compose from **opendetail-react** ([UI integration](/docs/ui-integration)). |
| **Vector DB?** | Not required for the default local index path. |
| **MDX with components?** | Text and headings matter for retrieval; exotic runtime components are not the retrieval unit. |
| **Hosted backend?** | Scaffolded client shape exists; managed backend is not shipped in this phase. |
