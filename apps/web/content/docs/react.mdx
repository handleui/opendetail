Use **`opendetail-react`** when you want OpenDetail in a React app. It gives you the browser state, the assistant components, and the wiring needed for the **Fastest** path. It also works well for the **Branded** path when you want your own layout but still want OpenDetail to manage retrieval and streaming.

If you are building a non-React client, use the transport client from `opendetail-client` or the Node engine in [`opendetail`](/docs/core).

## What this package is for

- **Fastest**: use the ready-made assistant components and keep your effort low.
- **Branded**: use the hook and primitives, then place them inside your own shell.
- **Not this package**: if you only want the transport client with no React state, use `opendetail-client`.

## The custom UI contract

OpenDetail keeps the frontend contract intentionally small.

Your UI sends:

```json
{ "question": "..." }
```

The backend responds as a stream of NDJSON events. The important event types are `meta`, `sources`, `images`, `delta`, `done`, and `error`.

The order matters: metadata can arrive first, sources and images can appear as the assistant grounds itself, `delta` carries streamed text, and `done` or `error` ends the turn. If the request cannot start, the backend returns JSON instead of NDJSON.

That is the main thing a custom UI needs to know. Everything else is presentation.

## `useOpenDetail`

`useOpenDetail` manages browser state for a React chat surface. It tracks the current draft, message list, request status, and errors while it talks to your `/api/opendetail` route.

<CodeBlock lang="ts">{`import { useOpenDetail } from "opendetail-react";

const {
  messages,
  question,
  setQuestion,
  submit,
  stop,
  status,
} = useOpenDetail({
  transport: { endpoint: "/api/opendetail" },
});`}</CodeBlock>

Use the hook when you want to own the layout but not the transport logic.

## Assistant components

The package also ships higher-level React pieces for the Fastest path:

- `AssistantSidebar` for a drop-in assistant panel
- `AssistantModal` for modal presentation
- `Thread`, `Composer`, `UserMessage`, `AssistantMessage`, `AssistantSources`, and `AssistantStatus` for smaller custom compositions

These pieces are convenience layers on top of the same backend contract. If you outgrow them, you can keep the contract and replace the UI.

## Source links

For docs sites, `renderNextSourceLink` from [`opendetail-next`](/docs/next) turns local citations into safe Next.js links and keeps remote URLs as regular anchors. If your docs site already has predictable page URLs, pair that with [`opendetail-fumadocs`](/docs/fumadocs) so citation targets stay on known pages.

## Styling

`opendetail-react` includes a default visual system. Use it as-is for Fastest, or treat it as a base layer for Branded if you want to change the look without changing the contract.

The important separation is this:

- changing tokens keeps you in the same system
- changing structure means you are really building a custom UI

That distinction helps keep the product easier to reason about for both adopters and maintainers.
