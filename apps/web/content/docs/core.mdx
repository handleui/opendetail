Use **`opendetail`** when you want the engine only. It reads **`opendetail.toml`**, builds **`.opendetail/index.json`**, and gives you the Node runtime that answers questions from that index. This is the **Headless** path, and it is also the backend layer behind the other paths.

## What ships in the package

| Area | Role |
| --- | --- |
| **Build** | Parse Markdown and MDX, chunk by headings, and write the index. |
| **Runtime** | Load the index, search it, call the model with grounded context, and return an answer. |
| **CLI** | Setup, build, doctor, and verification commands. |
| **Config** | `opendetail.toml` and related sidecar files. |

## When to use it directly

Choose `opendetail` when you want a custom server, a background job, a test harness, or an agent integration that does not need React or Next.js. If your app is already on Next.js, `opendetail-next` is the thinner route adapter. If you are only building a browser experience, `opendetail-react` handles the UI state on top of the same backend contract.

## Minimal config

OpenDetail reads **`opendetail.toml`** at the project root.

<CodeBlock lang="toml">{`version = 1

[[content]]
include = ["content/**/*.{md,mdx}"]
exclude = []
public_path = "/docs"
`}</CodeBlock>

## How the backend contract works

The runtime is intentionally narrow.

Your app sends a question:

```json
{ "question": "How do I configure public_path?" }
```

The engine uses the built index to find the most relevant content, builds grounded context, and asks the model to answer from that context. It then returns the answer along with sources, images, and fallback metadata.

When you stream, the response is NDJSON with events such as `meta`, `sources`, `images`, `delta`, `done`, and `error`. That is the contract your UI or agent should consume. You do not need to know how the index was built in order to use it.

## Fields

| Field | Required | Meaning |
| --- | --- | --- |
| `version` | yes | Must be `1` |
| `[[content]]` | yes | One or more tables: each has `include`, `exclude`, and `public_path` for a Markdown/MDX root |
| `media` | no | Rules for mapping local media references to public URLs |
| `fetch` | no | Optional model tools such as file search or web search |
| `knowledge` | no | Optional image metadata sidecar |

## Content roots

You can list multiple `[[content]]` blocks when different parts of the site live in different folders or under different URL prefixes. Set `public_path` to the public prefix for that tree, then OpenDetail will map each indexed file to the right citation URL.

## Media

Use `[media]` when docs reference local images such as `./hero.png`.

<CodeBlock lang="toml">{`[media]
include = ["content/**/*.{png,jpg,jpeg,webp,avif,gif,svg}"]
exclude = []
public_path = "/content-media"
`}</CodeBlock>

When `media` is configured:

- `http://` and `https://` URLs are returned as-is
- root-relative `/...` URLs are returned as-is
- local relative paths are mapped only if they match the media glob
- unsupported absolute schemes and protocol-relative URLs are ignored

## Fetch

Optional retrieval tools can be configured in `opendetail.toml` or passed at runtime. Use this when you want the model to reach beyond the indexed docs.

<CodeBlock lang="toml">{`[fetch.file_search]
vector_store_ids = ["vs_123"]
max_num_results = 8

[fetch.web_search]
allowed_domains = ["opendetail.dev", "platform.openai.com"]
search_context_size = "low"
`}</CodeBlock>

## Create an assistant

Use `createOpenDetail` from Node when you want direct access to the engine.

<CodeBlock lang="ts">{`import { createOpenDetail } from "opendetail";

const assistant = createOpenDetail();

const result = await assistant.answer({
  question: "How do I configure public_path?",
});`}</CodeBlock>

`answer()` returns the final text plus sources, images, and fallback state. `stream()` gives you the same result as a stream if you want to forward events to your own UI.

## Defaults

| Setting | Default |
| --- | --- |
| `model` | `gpt-5.4-mini` |
| `reasoningEffort` | `none` |
| `verbosity` | `low` |
| `store` | `false` |
| `promptCacheRetention` | `in-memory` |

## Fallback behavior

If lexical retrieval finds no indexed chunks, OpenDetail marks the result as fallback. That means the answer is still allowed to come from the model or configured tools, but the index did not supply a direct hit.

If nothing usable comes back, the safe product response is still a short, honest fallback message rather than a guessed answer.
