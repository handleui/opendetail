The **`opendetail`** package is the engine: it reads **`opendetail.toml`**, builds **`.opendetail/index.json`**, and exposes **`createOpenDetail`** for answering from that index. The CLI (`build`, `setup`, `doctor`) lives in the same package — see [**CLI**](/docs/cli).

## What ships in the package

| Area | Role |
| --- | --- |
| **Build** | Parse matched Markdown/MDX, chunk by headings, write the index (CLI or `buildOpenDetailIndex`). |
| **Runtime** | Load the index, lexical search, Responses API, stream NDJSON — [Create an assistant](#create-an-assistant). |
| **CLI** | Setup, index builds, doctor — [**CLI**](/docs/cli). |
| **Config** | `opendetail.toml` — [Fields](#fields). |

## When to use it directly

Use **`opendetail`** without Next or React for Node integrations (custom server, scripts, tests). For Next.js App Router, use [**opendetail-next**](/docs/next); for browser UI, use [**opendetail-react**](/docs/react) and [UI integration](/docs/ui-integration).

## Minimal config

OpenDetail reads **`opendetail.toml`** at the project root.

<CodeBlock lang="toml">{`version = 1

[[content]]
include = ["content/**/*.{md,mdx}"]
exclude = []
public_path = "/docs"
`}</CodeBlock>

## Fields

| Field | Required | Meaning |
| --- | --- | --- |
| `version` | yes | Must be `1` |
| `[[content]]` | yes | One or more tables: each has `include`, `exclude`, and `public_path` for a Markdown/MDX root |
| `media` | no | Rules for mapping local media references to public URLs |
| `fetch` | no | Optional OpenAI Responses API tools: `[fetch.file_search]` and/or `[fetch.web_search]` |
| `knowledge` | no | Path to optional `.opendetail/knowledge.toml` sidecar with `[[asset]]` titles/summaries for images |

## Content roots

You can list **multiple** **`[[content]]`** blocks when docs, marketing, or other MDX trees live under different folders **and** are served at different URL prefixes. Each row is one indexed tree. Set **`public_path`** to the **public URL prefix** for that tree (where those pages appear in the browser), for example `/docs` for documentation and `/` for site-root marketing. OpenDetail joins each file’s route segments after that prefix to build citation URLs (and, beyond the first root, tags chunks as `sourceKind: "page"` for UI styling).

## Media

Use `[media]` when docs reference local images such as `./hero.png`.

<CodeBlock lang="toml">{`[media]
include = ["content/**/*.{png,jpg,jpeg,webp,avif,gif,svg}"]
exclude = []
public_path = "/content-media"
`}</CodeBlock>

When `media` is configured:

- `http://` and `https://` URLs are returned as-is
- root-relative `/...` URLs are returned as-is
- local relative paths are mapped only if they match `[media].include`
- unsupported absolute schemes and protocol-relative URLs are ignored

If a doc references a local image that does not match the configured media globs, the build fails with a clear error.

## Fetch (OpenAI remote tools)

Optional Responses API tools — configure as **`[fetch.file_search]`** and/or **`[fetch.web_search]`** (native remote retrieval; no same-origin HTML scraping).

<CodeBlock lang="toml">{`[fetch.file_search]
vector_store_ids = ["vs_123"]
max_num_results = 8

[fetch.web_search]
allowed_domains = ["opendetail.dev", "platform.openai.com"]
search_context_size = "low"
`}</CodeBlock>

**`file_search`**

| Field | Required | Meaning |
| --- | --- | --- |
| `vector_store_ids` | yes | OpenAI vector store IDs |
| `max_num_results` | no | Limit from 1 to 50 |

**`web_search`**

| Field | Required | Meaning |
| --- | --- | --- |
| `allowed_domains` | no | Restrict search to listed domains |
| `search_context_size` | no | `low`, `medium`, or `high` |

Remote citations appear in `sources` with `kind: "remote"`.

## Knowledge sidecar

Optional **`knowledge = ".opendetail/knowledge.toml"`** with entries such as:

<CodeBlock lang="toml">{`[[asset]]
url = "/content-media/diagram.png"
title = "Pipeline overview"
summary = "Use when explaining the build step."
`}</CodeBlock>

## Create an assistant

Use **`opendetail`** core when calling from Node without Next.js helpers.

<CodeBlock lang="ts">{`import { createOpenDetail } from "opendetail";

const assistant = createOpenDetail();
`}</CodeBlock>

Then:

<CodeBlock lang="ts">{`const result = await assistant.answer({
  question: "How do I configure public_path?",
});
`}</CodeBlock>

`answer()` returns `text`, `model`, `sources`, `images`, `fallback`. `stream()` adds a `ReadableStream<Uint8Array>`.

## Assistant instructions

The OpenAI Responses API separates **`instructions`** (how the model behaves)
from **`input`** (sources + question). OpenDetail merges **harness** and **project**
text into a single `instructions` string:

- **Harness** (built-in): informational assistant behavior, grounding, citations, tools.
- **Project** (your files below): tone, priorities, product vocabulary — keep it short.

Loaded in order (project append):

1. `assistantInstructions` option
2. `assistantInstructionsPath` option
3. `.opendetail/OPENDETAIL.md`
4. `OPENDETAIL.md` at project root

If none exist, only built-in harness instructions apply.

## Remote retrieval

Configure via **`fetch`** in `createOpenDetail` options or **`[fetch]`** in `opendetail.toml` (`fetch.file_search`, `fetch.web_search`). Runtime options override the artifact when provided.

## Defaults

| Setting | Default |
| --- | --- |
| `model` | `gpt-5.4-mini` |
| `reasoningEffort` | `none` |
| `verbosity` | `low` |
| `store` | `false` |
| `promptCacheRetention` | `in-memory` |

## Important options

| Option | Meaning |
| --- | --- |
| `cwd` | Working directory for config and instruction files |
| `indexPath` | Custom path to the built index |
| `indexData` | Preloaded index artifact |
| `client` | Reuse an existing OpenAI client |
| `assistantInstructions` | Inline project guidance |
| `assistantInstructionsPath` | Custom instructions path |
| `promptCacheKey` | Explicit cache key override |
| `fetch` | Override `fetch` from the index (OpenAI tools) |

## Fallback behavior
